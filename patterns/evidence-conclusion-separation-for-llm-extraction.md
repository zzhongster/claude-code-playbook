# 证据-结论分离 — 大规模 LLM 抽取的可追溯架构

> 来源：《元始法则》三维世界模型工程：112 批次、4800+ 条证据、978 节点世界树（2026-07）
> 结局：全量构建后，20 章增量更新只需跑 2 个新批次 + 十来条新裁决，其余全是确定性脚本重放

## 问题

用 LLM 从长文本（小说/文档/日志）批量抽取结构化知识时，如果让模型**直接产出最终结论**（"世界树长这样"），会同时踩三个坑：

1. **不可追溯**——结论错了不知道是哪一章的哪句话导致的
2. **不可增量**——新内容进来只能全量重跑，或在旧结论上打补丁越打越歪
3. **判断被摊薄**——每个抽取 agent 都在顺手做合并/裁决，便宜模型不敢用，贵模型批量跑烧钱

## 架构：三层单向流

```
evidence/batch-*.yaml     ← LLM 抽取层（append-only，每条带原文引句+章节+kind+certainty）
        ↓
decisions.yaml            ← 裁决层（append-only，人/强模型只处理 merge 报上来的矛盾）
        ↓
merge.py → model/         ← 合成层（纯确定性脚本，随时可全量重放）
```

- **证据层只记录，不判断**：每条证据 = 原文短引句 + 出处章节 + 类型（存在/归属/方向/距离…）+ 确定性（stated/legend/speculation）。抽取 prompt 是纯契约执行，**可以放心用便宜模型**（本案例 Sonnet 跑抽取，成本约为主模型 1/4）
- **裁决层只处理冲突**：合成脚本检测到矛盾（归属冲突/疑似别名/环）时生成 pending 问题清单，裁决以 append-only 方式写入，带引句依据。先跑机械消解规则（链式归属、部位后缀），剩下的才值得花模型/人力
- **合成层幂等**：`merge` 任何时候全量重放，产出一致。改一条裁决→重放→立即看到全局影响

## 为什么增量便宜

断点锚 = **证据文件的存在性**（`evidence/batch-X.yaml` 在就是抽过了），配一个 watermark 记录水位。新增 20 章：抽 2 个新批次 → merge 报出 8 个新矛盾 → 裁决 → 重放。旧的 4700 条证据和 800 条裁决原封不动。

## 配套教训（都踩过）

- **alias 裁决的 names 字段只放"要被并掉"的名字**。把规范名或同名独立实体也写进去，会静默吞掉无辜节点（"岁月国"并入时 names 里多写了"岁月"，把一个独立的悬浮巨球实体也并没了）
- **历史回溯修辞会污染演变事件**："千年前浩劫等于灭了一次"这类句子会被抽成 destroyed，把现存实体标死。每轮增量抽查含"千年/万年/昔日/覆灭"的演变条目
- 抽取 agent 输出大 YAML 要**分段写盘**（先 Write 前半再 Edit 追加），单条回复有 token 上限，整文件一次吐会截断

## 反模式信号

- 抽取 prompt 里出现"如果和已有数据冲突请自行合并/取舍"——判断混进了抽取层
- 最终模型文件被 LLM 直接编辑过，重放脚本产出和仓库里的文件不一致
- 新数据进来的第一反应是"把旧结论发给模型让它更新"
