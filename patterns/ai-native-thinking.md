# AI Native 思维：LLM 做决策者，代码做后勤

## 核心发现

> 传统开发是"人写逻辑，AI 填空"；AI Native 是"AI 做研究员，代码做后勤"。

一次对比实验揭示了两种范式的本质差异：

| | 工具思维（Tool-Centric） | 智能体思维（Agent-Centric） |
|--|-------------------------|--------------------------|
| 做法 | 代码硬编排 → MAP 盲扫 55 个 URL → LLM 在最后一步提取文本 | LLM 理解任务 → 生成搜索策略 → 判断结果价值 → 定向抓取 → 迭代补全 |
| 结果 | **1 个产品**，16 credits，15 分钟 | **60 个产品**，5 credits，3 分钟 |
| 本质 | 把研究员变成流水线工人 | 让 LLM 扮演那个研究员 |

来源：工商银行贷款产品采集实验（2026-02-24）

## 五条设计原则

### 原则 1：LLM 做决策者，不做执行者

**错误**：代码决定抓哪些 URL，LLM 从抓回来的文本里提取。
**正确**：LLM 决定搜索什么、抓哪些页面、结果够不够、还需要什么。

人类研究员不会盲目打开 55 个网页然后逐个阅读。他会先搜索，看标题判断哪些值得点开，读完几篇后发现缺口，再补充搜索。LLM 应该做同样的事。

### 原则 2：搜索摘要是第一手情报

**错误**：搜索只是为了获取 URL 列表。
**正确**：搜索引擎返回的标题 + 摘要本身就包含大量结构化信息。

实验中，仅从 Brave Search 的摘要就能识别出 40%+ 的产品名。很多时候根本不需要抓取页面。

### 原则 3：精准的少量抓取 > 盲目的大量抓取

一个第三方聚合站整理的"产品大全"页面 = 50 个官网单产品页。LLM 知道这一点，代码不知道。

### 原则 4：多轮迭代，而非单次通过

**错误**：搜索一次 → 抓取一次 → 提取一次 → 结束。
**正确**：搜索 → 提取 → 检查覆盖度 → 发现缺口 → 补充搜索 → 再提取。

代码永远不会判断"三农服务类只有 1 个产品，不够"。代码只会执行完预设步骤就停止。

### 原则 5：利用 LLM 的领域知识

**错误**：假装 LLM 什么都不知道，只让它从抓到的文本里提取。
**正确**：LLM 已经知道银行产品有 6 大分类——利用这些知识指导搜索和验证。

LLM 的知识不是幻觉风险，而是研究效率的加速器。关键是用搜索和抓取来**验证和补充**知识，而非替代它。

## 三层思考模型

### 第一层：谁做决策？

问自己：代码里的 `if/else` 哪些其实是业务判断？

```python
# 这是代码该管的
if response.status_code == 429:
    time.sleep(backoff)

# 这应该交给 LLM
if "贷" in name or "融资" in name:  # ← 为什么不让 LLM 判断？
    products.append(name)
```

### 第二层：信息获取的效率金字塔

```
        /\
       /  \      LLM 自身知识
      /    \     (免费，有幻觉风险，用于指导方向)
     /------\
    /        \   搜索摘要
   /          \  (极低成本，标题+摘要=80%信息)
  /------------\
 /              \ 页面抓取
/                \ (有成本，只抓 LLM 认为值得的)
```

传统做法把金字塔倒过来——先抓 55 个页面（最贵），再让 LLM 提取（最后才用）。

### 第三层：迭代闭环——AI Native 的灵魂

传统流水线是开环：`输入 → 处理 → 输出`，做完就结束。

AI Native 是闭环：

```
目标 → 行动 → 结果 → 评估 → 够了？
                              ↓ 不够
                        调整策略 → 行动 → ...
```

**让 LLM 做那个"够了吗"的判断**，这是最关键的设计。

## 交叉验证：企业调研引擎实验（2026-03-12）

同一结论在另一个项目中被独立验证：

| 方案 | 架构 | 结果 | 问题 |
|------|------|------|------|
| Pipeline（多 Worker） | 代码编排搜索 → 廉价模型筛选 → 分 Worker 处理 | 找到 40 家，保留 **0 家** | Worker 无全局上下文，廉价模型过度过滤 |
| Skill（单对话） | LLM 全程决策，自行搜索-判断-迭代 | 找到 15 家，保留 **13 家** | — |
| Skill + 提示词优化 | 四阶段工作流 + 品类覆盖检查 | 找到 **21 家**，全部有效 | — |

关键发现（详见 [prompt-over-model](prompt-over-model.md)）：
- **提示词优化 > 换模型 > 改架构**
- 需要全局判断力的任务**不能拆成隔离的 Worker**

## 迁移自检：三个问题

无论做什么产品，问自己：

1. **代码里哪些 if/else 其实是业务判断？** → 交给 LLM
2. **流程是"做完就停"还是"做完先检查"？** → 加闭环评估
3. **在哪个环节才让 LLM 介入？能不能提前？** → LLM 介入越早，浪费越少

## 通用架构模板

```
┌─────────────────────────────────────────────────┐
│                   LLM Agent                      │
│                                                   │
│  ┌─────────┐  ┌──────────┐  ┌───────────────┐   │
│  │ 搜索规划 │→│ 结果判断  │→│ 覆盖度检查     │   │
│  │(生成query)│ (值不值得抓)│ (缺什么补什么)  │   │
│  └─────────┘  └──────────┘  └───────────────┘   │
│       ↕             ↕              ↕              │
│  ┌─────────────────────────────────────────┐     │
│  │           工具层（LLM 的手和脚）          │     │
│  │  🔍 Search API    📄 Page Fetch         │     │
│  │  🧠 LLM Extract   💾 Storage           │     │
│  └─────────────────────────────────────────┘     │
└─────────────────────────────────────────────────┘
```

适用场景：竞品分析、行业研究、政策追踪、供应商发现——一切"从互联网收集结构化信息"的任务。
