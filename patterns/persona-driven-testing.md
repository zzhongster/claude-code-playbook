# 角色扮演测试法（Persona-Driven Testing）

## 核心原则

> 不要从功能清单出发测试，从"一个真实用户的一天"出发测试。

传统测试：`按钮能点吗？接口返回 200 吗？字段校验对吗？`
角色扮演测试：`一个助贷员早上 8:30 打开手机，他第一件事想做什么？他能在 3 秒内做到吗？`

前者测功能是否"能用"，后者测产品是否"好用"。

## 方法：四步走

### 第一步：定义角色

不是"用户 A"，是一个有血有肉的人：

```markdown
### 我是谁？
我是一个助贷员（贷款中介），我不是银行的人。
我的工作是帮客户找到最合适的银行贷款产品，赚取佣金。
我同时服务几十个客户，每天要处理：
- 新客户进来（电话/微信/转介绍），要快速录入信息
- 给客户匹配贷款产品，出方案
- 跟踪银行审批进度，银行要补材料我得催客户
- 客户放款后，关注到期时间，提前帮他续贷
```

关键信息：**职业身份、技术水平、日常工具、工作节奏**。

### 第二步：写"我的一天"

```
8:30  打开手机看今天要做什么（晨报/待办）
9:00  给昨天新来的客户 A 录入信息、匹配产品
9:30  客户 B 的材料银行说不合格，要通知 B 重新准备
10:00 客户 C 打电话问"我的贷款批了没有"
10:30 新客户 D 打电话来，说"我做餐饮的，想贷100万"
11:00 给客户 A 发匹配方案
14:00 客户 E 的贷款 3 个月后到期，要提前规划续贷
16:00 整理今天的进件情况
```

这就是你的测试场景来源——不是凭空设计的，是从真实工作流里提取的。

### 第三步：场景化测试

每个场景是一个**故事 + 测试要点**：

```markdown
### 场景 A：电话来了 — 快速录入新客户

**故事**：正在打电话，客户说"我叫王伟，做建材批发的，公司名叫鑫达建材，
在成都，干了6年，月流水大概40万，想贷200万扩大仓库，有一套房，征信还行"。
我需要在通话中把这些信息快速记下来。

**测试要点**：
□ 从首页到"新增客户"需要几步？（≤2步为优）
□ AI 智能录入：把上面那段话粘贴进去，能识别出多少字段？
□ Tab 键能否在字段间跳转？顺序合理吗？
□ 录入完点"创建"后，能否立刻开始匹配？
□ 如果录入到一半来了另一个电话，返回后数据还在吗？
□ 移动端：键盘弹出后，表单会不会被挡住？
```

注意测试要点**从用户目标出发**（"快速记下来"），而非从功能出发（"表单组件是否渲染"）。

### 第四步：量化评分

#### 三级评分

- 🟢 符合期望或超预期
- 🟡 能完成但有摩擦（需要想一下怎么操作、多点了一步、等了一会）
- 🔴 无法完成，或严重影响效率

#### 五维度加权

| 维度 | 权重 | 说明 |
|------|------|------|
| 录入效率 | 25% | 新数据从 0 到录完要多久？ |
| 核心流程 | 20% | 主流程链路是否闭环？ |
| 跟踪管理 | 20% | 状态信息是否一目了然？ |
| 移动端可用性 | 20% | 手机上完成全部流程的舒适度 |
| 容错与引导 | 15% | 出错时提示清晰吗？新手知道怎么用吗？ |

权重根据产品类型调整——to C 产品移动端权重更高，to B 后台录入效率更重要。

## 配套：自动化

角色扮演测试不意味着纯手工。用 Playwright 把场景脚本化：

```javascript
// test-05-real-workflows.mjs — 场景 A：快速录入新客户
test('快速录入：AI 智能识别', async ({ page }) => {
  await page.goto('/customers/new');

  // 粘贴自然语言描述
  await page.fill('[data-testid="ai-input"]',
    '王伟，做建材批发的，公司叫鑫达建材，在成都，干了6年...');
  await page.click('[data-testid="ai-parse"]');

  // 验证识别结果
  expect(await page.inputValue('[name="name"]')).toBe('王伟');
  expect(await page.inputValue('[name="city"]')).toBe('成都');
  // ...
});
```

配合截图对比，每次迭代后自动回归。

## 什么时候跑

- 每次 UI/UX 相关改动后
- 每次新功能上线后
- 发版前必跑
- 大的数据结构变更后（可能影响页面渲染）

## 反模式

### "功能清单"测试

```
□ 登录功能正常
□ 列表页能显示数据
□ 表单能提交
□ 搜索能用
```

这种测试只能发现"坏了"，发现不了"不好用"。

### 开发者视角测试

开发者测试时总是走最短路径、输入完美数据、在最大屏幕上操作。而真实用户在手机上、打着电话、输入半句话、网络不稳定。**测试约束条件应该来自角色定义，不是开发者习惯**。
