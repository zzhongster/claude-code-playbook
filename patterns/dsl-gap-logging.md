# DSL Gap Logging：用日志驱动 DSL 扩展

## 核心发现

> 当 LLM 生成的表达式被 DSL 引擎拒绝时，不要只返回错误——记录下来。积累的失败日志就是 DSL 扩展的需求池。

在量化因子实验室项目中，LLM 将自然语言翻译为 DSL 表达式，但引擎只支持有限的运算符集合。LLM 频繁生成 `&`（逻辑与）运算符，但引擎不认识，导致所有复合条件因子全部报错。

修复很简单——给引擎加上 `&` 和 `|`。但问题是：**下一个缺失的运算符是什么？**

## 模式

在 DSL 校验失败时，记录一条结构化日志：

```python
# validate 失败时记录 DSL_GAP
if expression:
    err = engine.validate(expression)
    if err:
        valid = False
        validation_error = err
        logger.info(
            "DSL_GAP expression=%r nl=%r error=%r",
            expression, user_input, err,
        )
```

日志包含三个关键字段：
- **expression** — LLM 生成的原始表达式（看它想用什么语法）
- **nl** — 用户的自然语言输入（理解业务意图）
- **error** — 解析失败原因（定位具体缺失）

运营时用 `grep DSL_GAP logs/*.log` 定期扫描，按失败原因分类统计，高频缺失的运算符/函数批量添加。

## 为什么有效

1. **零成本采集**：不需要额外的反馈 UI 或数据库表，一行日志就够
2. **真实需求**：来自真实用户的真实尝试，不是臆测的功能清单
3. **有优先级**：出现频率天然就是优先级排序
4. **保留上下文**：同时记录 NL 和表达式，方便判断"用户到底想做什么"

## 适用场景

任何 **受限语言/DSL/API 对接 LLM 生成** 的场景：

| 场景 | Gap 日志内容 |
|------|-------------|
| 因子 DSL | 缺失的运算符、函数、字段 |
| SQL 生成 | 不支持的 SQL 方言/函数 |
| API 编排 | LLM 尝试调用的不存在的 endpoint |
| 代码生成 | 生成了目标语言不支持的语法 |
| 规则引擎 | 不在白名单中的条件/动作 |

## 反模式

- **只返回错误不记录** — 用户看到报错，开发者一无所知，同样的问题反复出现
- **记录太多信息** — 不需要完整请求上下文，三个字段足够决策
- **自动扩展 DSL** — 让 LLM 的输出自动成为新语法很危险，扩展必须是人工决策

## 实际效果

SAWSystem 因子实验室中，部署此模式后：
- 第一批发现 `&` 和 `|` 高频缺失 → 立即添加到引擎
- 后续可持续监控，按需扩展 DSL 能力集
- 翻译成功率随 DSL 能力扩展逐步提升

## 关联
