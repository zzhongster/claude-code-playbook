# 解码器版本矩阵 — 定位"同一份数据不同表现"的 bug

> 来源：声伴 MP3 坏静音帧追查（2026-07-04）
> 结果：305 个 MP3 的数字静音帧编码不规范，Chrome 149 新解码器整帧丢弃（时间轴 108s→91.6s），旧解码器容错输出静音

## 问题

解码/渲染类 bug 的表象是"同一份数据，有人正常有人异常"。直觉会引向数据传输（文件被改了？）或业务代码（播放逻辑错了？），但真正的分歧点可能在**解码器实现与版本**——这是最容易被遗漏的轴。

## 做法

把所有能拿到的解码路径摆成矩阵，同一份字节喂进去，记录输出：

| 解码器 | 时长 | 开头内容 |
|---|---|---|
| ffmpeg CLI | 108.04s | 静音 ✓ |
| CoreAudio (afinfo) | 107.95s | — |
| Chromium 145 decodeAudioData | 108.04s | 静音 ✓ |
| Chromium 145 audio 元素 | ~108s | 静音 ✓ |
| **Chrome 149 decodeAudioData** | **91.59s** | **音乐 ✗** |

矩阵一摆出来，分歧轴（版本 145→149）和一致轴（API 无关、平台无关）同时清晰。注意**每次只变一个轴**：我最初的对照是"我方 audio 元素 vs 用户方 decodeAudioData"，API 和浏览器两个变量同时在变，结论不可用；补齐矩阵后才可判读。

## 定位后的三板斧

1. **量化差异**：108.04−91.59 ≈ 630 帧 → 与文件中"重复次数最多的帧"出现 629 次精确吻合 → 丢的就是静音帧
2. **提取坏帧原型**：`Counter(所有帧).most_common(1)` 拿到字节级原型 → 用"原型×3 连续出现"做子串扫描，全库秒级定位受影响文件（240+65 个，全部同一渲染批次）
3. **实证修复变换**：候选变换各做一个样本文件，在**故障版本的解码器里**验证（重编码 libmp3lame 治愈；`-c copy` 重封装无效——坏帧被原样拷贝）

## 部署建议：覆盖目录优先于改源

修复 303 个文件时没有动 OSS 原始资产，而是服务端加"覆盖目录"：`/api/file` 先查 `audio-fixes/{id}-{field}.mp3`，命中即用，否则回落原路径。**加法式、可回滚、原始数据零风险**，比批量覆写生产资产安全一个量级。

## 经验

- 浏览器自动升级意味着这类 bug 会"无人改代码却突然爆发"，且按用户升级进度**渐进式蔓延**——首报用户往往被当成个例（本案例正是）
- 编解码器对不规范数据的**容错行为不属于契约**，新版收紧容错是常态；生成媒体文件的管线要用主流编码器（libmp3lame），别信"能播就行"
- Python 逐帧解析 4MB×几千文件会超时；提取原型后用 `bytes.find`（C 级子串搜索）扫描，快两个数量级
