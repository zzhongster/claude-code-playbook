# 模式：用 ES terms + min_doc_count 从全库数据驱动地生成"高频共享/占位值"黑名单

## 一句话结论

当一个字段里混着大量"被无数条记录共用的脏值"（占位电话 `13800138000`、占位邮箱 `123456@qq.com`、数据商填充的代理商联系方式），别靠手写小黑名单——用 ES `terms` 聚合 + `min_doc_count` 阈值，让**数据自己告诉你哪些值是共享的**：一个值出现在 N+ 条无关记录上，它就不可能是某条记录的专属真实值。关键是用 `min_doc_count`（服务端只返回超阈值的桶）而不是死拉大 `size`（会撞 `search.max_buckets` 上限、且把海量长尾也拉回来）。

## 场景

- 企业库/联系人库/商品库里某字段（phone/email/地址/税号）有数据商批量填充的占位或共享值
- 现状是手写几个已知占位值的小黑名单，覆盖不全
- 想数据驱动地识别"高频复用=脏"的值，但字段基数上亿，不能全量拉回本地统计

## 做法

```json
GET company_db/_search
{
  "size": 0,
  "aggs": {
    "shared": {
      "terms": {
        "field": "phone",           // keyword 字段, 或 email.keyword
        "min_doc_count": 100,        // ★ 只返回出现在 100+ 文档的值 = 共享/占位
        "size": 20000               // 上限, 配合 min_doc_count 后返回量可控
      }
    }
  }
}
```

- 落盘成静态黑名单 json（phones/emails set + 阈值 + 生成说明），消费方加载后 O(1) 判断。
- 消费侧**诚实抑制**：命中黑名单的值置 None，并在响应 warnings 里说明"N 个联系方式因全库高频共享(疑似占位)已抑制"——不是静默删，让调用方知道发生了什么。
- 附一个可复现的生成脚本，因为库在增长，黑名单需定期重跑刷新。

## 阈值怎么定（关键判断）

- 真实专属值的 doc_count ≈ 1（只在自己名下）或个位数（少数关联实体）。
- 阈值取几十~上百：既盖住明显共享（几千上万家共用的），又不误伤集团总机（十几家子公司真共用）。
- **误伤在语义上其实是对的**：某代理公司的邮箱被 100 家客户共用——对这 100 家客户画像而言，那不是客户自己的邮箱，抑制它正确。阈值保守（≥100）时，"真实专属值被误抑制"在逻辑上几乎不可能（一个号不可能真属于 100+ 家不同公司）。

## 坑

- `terms` 死拉大 `size`（想一次拿全部高频值）会撞 `search.max_buckets`（默认 65536），报错或截断。用 `min_doc_count` 让服务端先过滤，返回量自然可控。
- `min_doc_count` 聚合有 `doc_count_error_upper_bound`，边界值可能漏收，但方向保守（偏向不误伤），可接受。
- 静态黑名单会随库增长漂移——生成脚本要留着，接定期刷新（或至少记 backlog）。

## 教训

- "手写黑名单"永远追不上数据；"高频=共享"这个信号本身就在数据里，让聚合把它捞出来。
- trade-agent-data（2026-07）实测：company 库 phone/email 各聚合出 2.3万/1.7万个 doc_count≥100 的共享值（top 的占位号被 8 万家公司共用），enrich_company 接入后能诚实抑制假联系方式，直接提升画像可信度。
