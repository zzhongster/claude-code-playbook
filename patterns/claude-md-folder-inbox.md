# CLAUDE.md Folder Inbox：用 CLAUDE.md 把文件夹变成自整理收件箱

## 核心原则

> 给一个"杂物文件夹"配一份 CLAUDE.md，把分类规则、命名规范、解析坑点固化进去，文件夹就变成了"丢文件即处理"的自动收件箱。

用户侧的操作成本降到最低：把文件丢进顶层，对 Claude 说一句"整理"（甚至只是打开这个目录开始对话），剩下的读取、分类、重命名、归档全部按既定规则自动完成。

## 适用场景

- 发票/报销凭证整理（本 pattern 的来源场景）
- 截图、下载文件、论文 PDF 等任何"持续流入、需要按内容归档"的文件夹
- 共同点：**文件的正确归宿由其内容决定**，而内容需要解析（PDF 文本、图片 OCR、文件名规律）

## 结构设计

```
收件箱文件夹/
├── CLAUDE.md                 ← 整理规则（本体）
├── <mm-dd>/                  ← 一次整理 = 一个批次文件夹，以整理当天日期命名
│   ├── 分类A/
│   └── 分类B/
└── （顶层散落的文件 = 待整理队列）
```

三个关键设计决策：

### 1. "顶层 = 待办队列"约定

不需要专门的 inbox/ 子目录。规则写成：**顶层除 CLAUDE.md 和日期批次文件夹外，一切都是待整理的新文件**。用户丢文件时零思考。

### 2. 按"处理日期"分批，而不是按内容日期归档

每次整理产出一个 `mm-dd/` 批次文件夹（处理当天的日期）。好处：

- 一次报销 = 一个批次，物理边界和业务边界一致
- 不会动到历史批次，整理操作天然幂等、可回溯
- 文件名内部再用业务日期（乘车日/入住日），两层日期各司其职

### 3. 命名规则取"业务发生信息"，不是文件元数据

发票场景的具体规则（写进 CLAUDE.md 的原文）：

- 日期一律 `mm-dd`，取**业务发生日期**（乘车日期/入住日期），不是开票日期
- 车票：`mm-dd 始发-终点`（站名去"站"字）
- 住宿：`mm-dd 酒店名`——门店名取**备注栏**，因为销售方名称往往是"XX酒店管理有限公司"而非门店名
- 新类型（餐饮、打车）出现时新建分类文件夹，汇报中说明分类依据

这样文件名直接构成行程/消费的时间线，`ls` 一眼对账。

## 关键：把解析坑点也写进 CLAUDE.md

规则文件不只记"做什么"，还要记**这类文件的解析陷阱**，否则下次会话会重新踩一遍：

```markdown
## 解析注意事项
- 部分铁路发票使用全角字符（「２０２６年０４月２６日」），解析前先做 NFKC 归一化
- 乘车日期行的特征：日期后跟「HH:MM开」；开票日期行没有时刻，勿混淆
- 同批次重名时加后缀（如 03-19 宁波-杭州南-2）
- 解析失败的文件留在顶层不动，汇报中单独列出
```

最后一条是**兜底规则**：无法判断的文件不猜、不动，留在队列里等用户指示。自动化系统必须有"不确定时停下"的出口。

## 与 Living CLAUDE.md 的关系

这是 [living-claude-md](living-claude-md.md) 的一个特化：那篇讲"CLAUDE.md 跟代码同步"，这篇是"CLAUDE.md 跟**工作流**绑定"——规则就住在它管辖的文件夹里，新增分类、新发现的解析坑都回写到同一份文件，文件夹自带说明书。

## 实施清单

1. 和用户确认：分类维度、命名格式、批次粒度（一句"复述一下你的理解及设计"很有用）
2. 写 CLAUDE.md：目录结构图 + 整理流程 + 命名规则 + 解析坑点 + 兜底规则
3. 把已整理的存量文件按新结构归位，作为第一个批次
4. 之后每次整理若遇到新类型/新坑，**先更新 CLAUDE.md 再干活**
