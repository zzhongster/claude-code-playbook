# 开发计划审计

## 核心原则

> 开发计划写完不等于写对。在动手之前，用多维审计发现遗漏、冲突和盲区。

开发计划的常见问题不是"写得不够细"，而是"有整块东西被遗漏了"。审计的目标是在写第一行代码之前，把这些洞找出来。

## 实战案例：四维审计

一个量化交易系统的开发计划（v1，73 项任务，6 个 Sprint），经过四维审计后：

| 审计维度 | 发现的问题 |
|---------|-----------|
| **任务完整性** | 7 个功能项完全遗漏（止损倒计时、因子禁用工具、实盘 vs 回测对比等） |
| **结构一致性** | 编号冲突（P1 的 #13 和 P0 的 #13）、总数声称 86 实际 91、两个任务描述重叠 |
| **跨切面覆盖** | 6 个维度几乎空白：测试(0/10)、CI/CD(2/10)、CLAUDE.md 同步(0/10)、移动端(2/10)、安全(3/10)、性能 SLO(3/10) |
| **前置依赖** | 数据层需要重构（7 个 DB 合并）、前端无设计稿需要先做原型 |

审计后计划从 73 项扩展到 ~110 项，新增了 Sprint 0（数据层重构 + UI 原型）。**如果不做审计，开发到中期才会发现这些问题，代价大 10 倍**。

## 审计框架

### 维度 1：任务完整性

> 计划里有没有整块功能被遗漏？

方法：
- 从用户旅程出发，逐步骤检查是否有对应任务
- 从系统架构出发，逐模块检查是否有对应任务
- 对照竞品/行业标准，检查是否有功能缺失

常见遗漏：
- 错误处理和异常路径（只规划了 happy path）
- 数据迁移和兼容性（新旧版本过渡）
- 监控和可观测性
- 权限和多租户

### 维度 2：结构一致性

> 计划本身有没有自相矛盾？

检查项：
- [ ] 编号是否唯一？（跨 Sprint 不冲突）
- [ ] 总数声称值是否准确？
- [ ] 有没有两个任务描述同一件事？
- [ ] 依赖关系是否正确？（A 依赖 B，但 A 排在 B 前面）
- [ ] 优先级排序是否合理？（高优依赖低优？）

### 维度 3：跨切面覆盖

> 有没有"横穿所有模块"的关注点被忽略？

对每个跨切面维度打分（0-10）：

| 跨切面 | 检查什么 |
|--------|---------|
| **测试** | 每个功能有对应的测试任务吗？单元/集成/E2E 覆盖？ |
| **CI/CD** | 自动构建、自动部署、环境管理？ |
| **文档同步** | CLAUDE.md、API 文档、用户指南随代码更新？ |
| **移动端/响应式** | 每个页面都考虑了小屏适配？ |
| **安全** | 认证、授权、输入校验、敏感数据处理？ |
| **性能** | 有 SLO 目标吗？大数据量场景测试过吗？ |
| **可观测性** | 日志、监控、告警？出了问题能定位吗？ |
| **无障碍** | 键盘导航、屏幕阅读器、色彩对比度？ |

经验法则：**跨切面任务占总工作量的 15-20%**。如果计划里几乎没有，说明被遗漏了。

### 维度 4：前置依赖

> 有没有"不先做就没法开始"的事情被忽略？

常见前置依赖：
- **数据层重构**：现有数据结构能支撑新功能吗？要不要先迁移？
- **设计/原型**：有设计稿吗？没有就先做原型，冻结后再编码
- **第三方服务**：API 申请、账号开通、沙箱环境
- **团队技能**：有没有需要学习的新技术？

## 审计流程

```
1. 写完开发计划 v1
2. 换一个视角审计（让 Claude 用另一个会话审查，或自己换个角色）
3. 按四个维度逐一检查，记录发现
4. 评估每个发现的影响（阻塞？延期？可以后补？）
5. 修订计划，标注新增/修改的任务和来源
6. 更新总数和依赖关系
```

**关键**：审计应该在 Sprint 0 之前完成，不是开发中途。

## 让 Claude 做审计

```
审查这份开发计划，从四个维度检查：

1. 任务完整性：有没有功能被遗漏？（对照用户旅程和系统架构）
2. 结构一致性：编号冲突？总数准确？任务重叠？依赖关系正确？
3. 跨切面覆盖：测试/CI/安全/性能/文档/移动端，每个维度打 0-10 分
4. 前置依赖：有没有必须先做才能开始的事情？

列出所有发现，按影响程度分级：🔴 阻塞 / 🟡 影响效率 / 🟢 可以后补
```

用另一个 Claude 会话（干净上下文）审计效果更好——避免对自己写的计划"视而不见"。
