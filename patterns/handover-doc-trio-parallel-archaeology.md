# 系统移交文档三件套 + 并行 Agent 考古 — 让接手人和他的 AI 各读各的

> 来源：trade-ai 邮箱报告移交实战（2026-07-16）
> 场景：整条业务链路移交给新维护者（人 + 他的 Claude Code），历史文档大量过时

## 问题

系统移交时最常见的失败模式：
1. 只写一份文档，既想给人看又想给 AI 看，结果两边都不好用
2. 直接把历史文档打包丢过去，里面混着三代架构的过时描述，接手人按旧文档改代码
3. 写新文档时照抄旧文档/README，把过时信息二次固化（实战发现：**连模块自己的 README 红线描述都过时了**——引用的 Issue 早已关闭解决）

## 模式

### 1. 三件套按受众分离

| 文档 | 受众 | 写法差异 |
|---|---|---|
| 业务说明 | 接手人（人类） | 讲角色/流程/产出/误区，界面视角，禁术语裸奔；开头先澄清最容易误解的名词 |
| 技术实现 | 接手人的 Claude Code | 路径+符号名可直接 grep；开篇「开工前必读的 N 个事实」（分支状态/命名陷阱）；**废弃代码标注清单**（只标不删）；行号注明快照日期 |
| 历史文档索引 | 两者 | 每份旧文档标 ✅🟡❌ 时效 + 一句话理由；给「判定坐标系」（见到哪些字眼即为废弃实现）；给最短上手路径 |

关键：技术文档为 AI 写时，「防误导」比「全覆盖」重要——AI 自己会 grep 细节，但不会怀疑文档里的陷阱（如 main 分支落后生产分支 501 commit、同名目录毫无关系）。

### 2. 并行 Agent 考古（4 个正交方向）

一次派 4 个 Explore agent 后台并行，各给独立提示词：

- **后端链路**：路由 → 服务 → 数据模型 → 外部依赖的完整调用链
- **前端链路**：页面组件 → API 调用层 → 构建部署机制
- **文档盘点**：全部相关历史文档 + 逐份时效判断（给它当前架构作参照系）
- **版本演进考古**：git log 里程碑 + 新旧代码共存清单 + 分支状态

正交拆分的收益：互不重叠、单 agent 上下文够深、主会话只做交叉校验。主会话同时自己读核心入口文件（README/模型/路由列表）建立判断基准，agent 结论回来才有能力质疑。

### 3. 时效实证 — 不信任何文档，包括代码里的 README

| 断言类型 | 验证手段 | 实战抓到的坑 |
|---|---|---|
| 「功能 X 可用」 | grep 配置项 + 前端调用点 | send-email 端点存在但 SMTP 零配置、前端零调用 = 死路径 |
| 「限制 Y 仍存在」 | gh issue view 查关联 issue 状态 | README 写「暂不可导出」，其 issue 两个月前已关闭 |
| 「存量数据 N 份」 | 直接 SQL 数 | 凭印象写"1000+"，实测 46,710 |
| 「文件/符号存在」 | 批量 `[ -e ]` + grep -c | 38 个引用路径逐一验证 |

### 4. QA 复核脚本化

写完文档后跑三类批量验证，而不是通读一遍"感觉没问题"：
```bash
# 路径存在性
for p in <文档中全部路径>; do [ -e "$p" ] && echo OK || echo MISSING; done
# 关键符号断言
grep -c "def check_compliance" <file>   # 每个技术断言至少一个 grep 命中
# 数字实测
docker exec db psql -c "SELECT data_origin, count(*) FROM ... GROUP BY 1"
```

## 反直觉点

- **旧文档的价值不在内容在索引**：46 份历史文档里真正必读只有 6 份，其余的价值是"标记为过时防止误读"
- **数字是最容易错的**：流程/架构描述有 agent 交叉验证兜底，但数量级数字只有 DB 实测能兜底
- **「只标不删」是移交期纪律**：废弃代码清单让接手人有全图，删除动作留给他接手后自己做
