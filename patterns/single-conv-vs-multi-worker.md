# 单对话 vs 多 Worker：什么时候该拆，什么时候不该拆

## 一句话结论

需要全局意识的任务（搜索、判断、筛选）用单对话；独立无依赖的并行任务（抓取 N 个 URL）才拆成多 Worker。

## 场景

当你需要用 LLM 处理复杂任务，面临以下决策：
- "要不要拆成多个子 Agent 并行？"
- "拆了会不会更快更便宜？"
- "单对话上下文会不会溢出？"

## 详细说明

### 两种架构

```
单对话模式：
  一个 LLM ──→ 搜索 ──→ 分析 ──→ 补搜 ──→ 判断 ──→ 输出
  （全局上下文贯穿始终）

多 Worker 模式：
  Worker A ──→ 搜索关键词组 1 ──→ 结果
  Worker B ──→ 搜索关键词组 2 ──→ 结果  ──→ 合并 ──→ 判断 ──→ 输出
  Worker C ──→ 搜索关键词组 3 ──→ 结果
  （每个 Worker 孤立上下文）
```

### 多 Worker 的隐性代价

表面上：更多并行 = 更快 + 更多结果。实际上：

1. **丢失全局意识**：每个 Worker 不知道其他 Worker 找到了什么，无法"还缺什么品类"
2. **搜索偏科**：不同 Worker 可能重复搜同一方向，同时遗漏某些方向
3. **判断失准**：独立的 FilterWorker 只看到数据，没有调研全程的意图理解
4. **工具浪费**：309 次搜索 < 30 次精准搜索（更多 ≠ 更好）
5. **合并困难**：去重、冲突解决、优先级排序都需要额外逻辑

### 判断标准

| 任务特征 | 推荐架构 | 原因 |
|---------|---------|------|
| 需要"还缺什么"的判断 | 单对话 | 需要全局视野 |
| 需要根据前一步结果调整策略 | 单对话 | 有依赖关系 |
| 需要理解用户意图做筛选/判断 | 单对话 | 意图是全局的 |
| 处理 N 个独立对象（URL/文件/企业） | 多 Worker | 无依赖，可并行 |
| 每个对象的处理逻辑相同 | 多 Worker | 适合并行 |
| 对象间不需要相互参照 | 多 Worker | 无全局依赖 |

### 混合架构（推荐）

最佳实践是**混合使用**：

```
阶段 1：单对话搜索（需要全局意识）
  → 一个 LLM 做广搜 + 追线索 + 品类覆盖检查 + 去重
  → 输出：20 家企业清单

阶段 2：多 Worker 并行 enriching（独立无依赖）
  → 20 个 Worker 各自深度调研一家企业
  → 每个 Worker 只需要企业名 + 官网 + 字段列表

阶段 3：单对话判断（需要全局意识）
  → 一个 LLM 看到所有企业的完整数据，做最终筛选和排序
```

## 数据支撑

实验详情见 [2026-03-12 Skill vs Pipeline](../experiments/2026-03-12-skill-vs-pipeline.md)

| 指标 | 单对话 (Qwen) | 多 Worker (Pipeline) |
|------|-------------|---------------------|
| 搜索次数 | 27 | 309 |
| 发现企业 | 22 家 | 40 家 |
| 保留企业 | 22 家 | **0 家** |
| 品类覆盖 | 7/7 子品类 | 3/7 子品类 |
| 费用 | $0.61 | $4.17 |

多 Worker 搜索量是单对话的 11 倍，但品类覆盖只有 43%，最终全部被淘汰。

## 适用范围

- **适用于**：搜索调研、内容审核、数据分类、报告生成等需要理解全局的任务
- **不适用**：大规模数据处理（如批量翻译 1000 篇文章、批量抓取 500 个网页）这类天然并行的任务
