# 飞书 vs 企业微信：机器人推文件的选型

## 一句话结论

> **只要"往外推文件/通知"** → 企业微信**群机器人 webhook** 最省事（无 token、无 IP 白名单、无审批）。
> **要"私聊指定个人 + 双向收命令 + 还跑在动态 IP 的本机上"** → 飞书更灵活（长连接事件订阅、不强制 IP 白名单）。
> 选型的本质是：**平台按"凭证能造成多大破坏"决定要不要加网络层锁**，而你的部署形态（有没有固定公网 IP）决定哪条路顺手。

## 场景

你要让一个跑在本机 / 服务器上的脚本或 agent，把生成的文件（调研报告、构建产物、对账单）自动推送给人。三条主流路径：

| 维度 | 企微·群机器人 webhook | 企微·自建应用 | 飞书·自建应用 |
|---|---|---|---|
| 鉴权 | 仅一个 webhook key | corpid+secret 换 access_token(2h，需缓存) | app token |
| **IP 白名单** | **无** ✅ | **强制「企业可信IP」**（只认固定 IP） ❌ | 不强制（可选） |
| 审批 / 发版 | **无** ✅ | 建应用+配可见范围（无发版审核） | **scope + 发版本 + 管理员审核** |
| 发文件 | 支持 file ≤20M | 支持 file ≤20M | 支持（需 im:resource:upload） |
| **私聊指定个人** | ❌ 只能发到机器人所在群 | ✅ touser 精准发 | ✅ 按 open_id/email 直发 |
| 收事件/双向 | ❌ | 推到你的**公网回调 URL**（要服务器） | **长连接订阅**（客户端往外拨号，免公网回调） |
| 配置复杂度 | 极低 | 中（可信IP 是坎） | 高（权限发版是坎） |

## 详细说明

### 为什么企微群机器人没有 IP 白名单、自建应用却强制

**关键洞察：平台按凭证的"破坏半径"上锁。**

- 群机器人 `key` 的权限范围极小——**只能往它所在的那一个群发消息**。泄漏了顶多刷那个群，删了重建即可。破坏半径小 → **不上网络锁**。
- 自建应用的 `access_token` 能给**全可见范围**发消息、读数据，破坏半径大 → 强制「企业可信IP」：即使密钥泄漏，请求不来自登记的服务器 IP 也被拒（错误码 60020）。这是网络层的"第二把锁"（纵深防御）。
- 只认 IP、不认域名：校验发生在服务端看请求的**来源 IP**；允许域名等于让攻击者随意指向，失去意义。

→ 推论：**自建应用对"动态公网 IP 的本机"不友好**（家用/办公宽带 IP 一变就 60020，且无法用域名/DDNS 绕过），它是为"企业自有服务器 + 固定 IP"设计的。

### 为什么飞书对"本地 agent / 无服务器"更友好

- 飞书**不强制 IP 白名单**（提供但可选），把强制关卡放在**权限层**（scope 细分 + 发版本 + 管理员审核）。
- 飞书有**长连接事件订阅**（如 `lark-cli event +subscribe`）：**客户端主动往飞书建一条常连接**，事件顺着连接推回来，**不需要公网回调 URL、不需要固定 IP、能穿 NAT**。这让"跑在笔记本上、动态 IP、要双向交互"的 agent 成为一等公民。
- 企微的事件相反：**推到你配置的公网回调地址**，默认假设你有公网可达的服务器。

### 各自的"坎"

- **飞书**：权限改了要发版本经管理员审核才生效；文件权限 `im:resource:upload` 要加在**应用身份**而非用户身份——见 [反模式](../anti-patterns/feishu-bot-file-upload-scope-gotchas.md)。
- **企微自建应用**：「企业可信IP」对动态 IP 是真坑，且无"程序化改可信IP"的官方接口；动态 IP 想用，得给脚本配一个**固定出口 IP**（云主机/跳板机代理）。
- **企微群机器人**：几乎没坎，唯一限制是只能发群——变通法是**建一个只放目标人的小群当"收件箱"**。

## 选型建议（决策树）

1. **只推不收，且"发到一个群"能接受** → 企微群机器人 webhook（最省事，公司主用企微时尤佳）。
2. **要私聊指定个人 / 要双向（在 IM 里发命令→跑→推回） / 跑在动态 IP 的本机** → 飞书自建应用（P2P 按 email/open_id 直发；长连接收事件）。
3. **必须用企微私聊个人** → 企微自建应用，但先解决固定出口 IP，否则持续 60020。

## 数据支撑 / 实例

- **deep-research-report（2026-06）**：出海调研报告交付。最终用**飞书自建应用 P2P 按 email 私聊直发 .xlsx**（无需建群），跑在动态 IP 的 Mac 上，工作正常。
- **BookWiki / novel-ingest-agent**：用**飞书长连接** `lark-cli event +subscribe` 实现"手机点卡片按钮审批 → 本地 agent 收到回调 → resume"的双向闭环，**全程无公网服务器**——印证飞书对本地 agent 的友好。
- **企微**：群机器人 `upload_media`（type=file）→ `send`（msgtype=file），≤20M、media_id 3 天有效、无 token/无 IP 白名单（官方文档 path/91770）；自建应用强制可信IP（错误码 60020，path/90313）。此为官方文档调研，未在生产项目落地。

## 适用范围

- 适用于：单人/小团队把"本机或单台服务器生成的文件"推给 IM。
- 不适用于：高并发企业级消息中台（要另算频率限额、消息模板、合规留痕）。
- 注意：webhook key / app secret 都是密钥，存 600 文件、勿入 git；明文分享过的应及时在后台重置。
