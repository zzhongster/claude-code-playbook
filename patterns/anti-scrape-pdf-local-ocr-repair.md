# 反爬 PDF 本地 OCR 修复管线 — 显示正常、复制乱码的零成本解法

> 来源：《元始法则》小说 wiki 工程，60+ 章 PDF 原文修复（2026-07）
> 结局：零 API 成本找回全部正文，含旧抓取缺失的章节和被合并的双章

## 问题

某类反爬 PDF 用**字体子集混淆**：渲染显示完全正常，但复制/`pdftotext` 提取出来是乱码（字符码点被重映射）。直接喂给 LLM 或抽取管线会产出垃圾，且这种污染**不会报错**——下游只会看到"内容质量差"。

## 判定：先跑乱码检测，别靠肉眼

中文文本两个便宜的启发式，任一命中即判乱码：

- **"的"字频 < 1.5%**（正常中文叙事文本"的"占 3~4%，重映射后必然崩）
- **CJK 扩展 A 区（U+3400-4DBF）字符占比 > 0.5%**（正常简体文本几乎不用扩展 A，重映射常落进这个区）

抽样每章跑一遍，能精确圈出污染范围（本案例：1031 章之后全部中招，之前干净）。

## 管线：渲染回图像，本地 OCR 认回来

既然**显示是正常的**，渲染成图再 OCR 就绕过了字体混淆，且 macOS 全程本地、零 API 成本：

1. **Swift PDFKit 3x 渲染** + **Vision framework OCR**（`zh-Hans`，`VNRecognizeTextRequest` accurate 模式）——3x 分辨率是准确率关键，1x 会大量错字
2. **重组脚本**按章节标题正则切分回章文件；章号加范围过滤（如 `900 < num < 1200`）防 OCR 误读的数字被当成章标题
3. **合体章处理**：两章被排版进同一页序列时，按已知字数比例切分（本案例 3253:3105 恢复了被吞的一章）
4. **修复后复跑第 2 步的乱码检测复验**——OCR 也可能整页失败，必须闭环

## 配套要点

- OCR 前先 `ls` 对齐页数与章数预期，页数异常往往意味着缺章/多章
- 重组时打印每章字数表，人眼扫一遍异常值（过短=切分错，过长=合体章）
- 修复产物与原始抓取物**分目录存放**，保留可追溯性

## 反模式信号

- 直接把 `pdftotext` 输出喂给下游，没有任何文本健康度检查
- 用云端 OCR API 处理大批量本地文件（Vision/PDFKit 在 mac 上免费且质量足够）
- OCR 完不复验，默认"跑完=修好"
