# 决策记录系统（DDR）

## 核心原则

> 讨论会过去，代码会变化，但决策的原因必须留下来。

项目越复杂，越容易出现"这段代码为什么这样写？"→ 没人记得 → 不敢改 → 技术债累积。决策记录系统解决的就是这个问题。

## 实践：D 编号系统

来自一个 AI Native ERP 项目（15+ 个设计决策，16 个微服务），在 `docs/design/decisions.md` 中集中管理：

```markdown
# 设计决策记录 (Design Decision Records)

## 状态标记
- ✅ 已采纳
- 🔄 讨论中
- ❌ 已否决
- 🔀 被替代（注明被哪条替代）

## D001: 卡片生成三级架构
**状态**: ✅ 已采纳
**日期**: 2026-02-20
**背景**: 需要将 AI 生成的业务数据渲染为前端卡片...
**方案**: 意图识别 → 模板选择 → 数据填充三级架构
**理由**: 比端到端生成 HTML 更可控、可缓存、可测试

## D002: 意图→模板确定性渲染
**状态**: ✅ 已采纳
...
```

### 使用规则

1. 每条决策有**唯一编号**（D001, D002...），讨论中用 "参见 Dxxx" 引用
2. 新讨论产生方案时先记 🔄，确认后改 ✅，否决标 ❌/🔀
3. **单一事实来源**：所有设计决策只在这一个文件里找

### 在对话中引用

```
这个问题在 D007（权限感知 UI）里讨论过了，当时的结论是...
```

Claude 可以直接读 decisions.md 获取历史上下文，避免重复讨论。

## 变体：ADR（Architecture Decision Records）

另一个项目用了更结构化的 ADR 格式，每条决策一个文件：

```
.design-tracker/decisions/
├── 0001-smartchat-保持工具型对话.md
├── 0002-新增客户表单添加企业工商查询功能.md
└── 0003-新增客户表单添加AI自然语言录入功能.md
```

每个文件的格式：

```markdown
# ADR-0001: SmartChat 保持工具型对话

## 状态
已采纳 (2026-03-03)

## 背景
用户提出是否要将 SmartChat 升级为全功能 AI Agent...

## 决策
保持工具型对话，不做 Agent 化。

## 理由
1. 工具型对话可预测、可测试
2. Agent 化的错误成本高（金融场景）
3. 当前用户需求用工具型就能满足

## 后果
- SmartChat 只响应明确意图，不做开放式对话
- 需要为每种意图编写模板
```

## 两种格式怎么选

| | 单文件 D 编号 | 多文件 ADR |
|--|-------------|-----------|
| 适合 | 设计阶段频繁讨论，决策密集 | 架构决策少而重，每条都需要详细记录 |
| 优势 | 一个文件看全貌，方便交叉引用 | 每条决策完整自包含，适合 git blame |
| 劣势 | 文件会越来越长 | 文件多了不好全局浏览 |
| 示例 | 交互设计决策（D001-D015） | 架构决策（ADR-0001 ~ 0003） |

**建议**：一个项目只用一种。设计密集期用 D 编号，进入稳定开发期后切 ADR。

## 和 Claude Code 的配合

在 CLAUDE.md 中声明：

```markdown
## 决策记录
- 所有设计决策记录在 docs/design/decisions.md
- 新讨论产生方案时先记 🔄，确认后改 ✅
- 讨论中引用已有决策用 "参见 Dxxx"
```

这样 Claude 在做设计决策时会自动：
1. 先查 decisions.md 看有没有相关历史决策
2. 新决策自动追加到文件末尾
3. 讨论中引用已有编号
