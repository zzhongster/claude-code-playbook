# 飞书机器人推文件：权限加在了"用户身份"，且没发版本

## 症状

让飞书自建应用的机器人（tenant_access_token / bot 身份）自动推送文件：

- **文本消息能发**（`im/v1/messages` msgtype=text 成功）；
- **文件上传却一直失败**，`im/v1/files` 报 `99991672`：

  > Access denied. One of the following scopes is required: [im:resource:upload, im:resource]. **应用尚未开通所需的应用身份权限** … token_type=tenant

- 反复"在权限页申请了 im:resource"、清 token 缓存、甚至"重新发布"都没用，错误一字不变。

## 根因（两个独立的坑叠加）

### 坑 1：权限加错了"身份 tab"

飞书应用权限分两个互不相通的 tab：

- **应用身份权限**（`tenant_access_token`，即 bot 自动执行）
- **用户身份权限**（`user_access_token`，需登录用户授权后才生效）

`im:resource`（获取与上传图片或文件资源）在两个 tab 下都能搜到，很容易**被加到「用户身份权限」**。但 bot 自动推送走的是**应用身份**，加在用户身份下对它**完全无效**。

> 迷惑点：文本能发、文件不能发 —— 因为 `im:message` 恰好加在了应用身份下，而文件权限只加在了用户身份下。这个"半通"现象会让人误判成"权限延迟/缓存问题"。

### 坑 2：加了权限 ≠ 生效，必须发版本 + 管理员审核

飞书自建应用**权限变更必须「创建版本 → 发布 → 企业管理员审核通过」才对线上生效**。只在权限页"申请/添加"而不发布新版本，线上跑的仍是旧版本的权限集。

关键机制：**`tenant_access_token` 不绑定 scope**，权限是**按"线上已发布版本的权限集"在每次 API 调用时实时校验**。所以——

- 换一个**现取的新 token** 复现同样的 99991672；
- 清 lark-cli / 本地 token 缓存也没用；

→ 这两点能**确诊"是服务端权限没生效，不是客户端/token 问题"**，不用再在代码侧瞎试。

## 修复

1. 在 **「应用身份权限」tab**（不是用户身份）添加 **`im:resource:upload`（上传文件V2）** 或 `im:resource`。
2. **创建版本 → 发布 → 管理员审核通过**（管理后台审批；不是开发者后台点一下就行）。
3. 验证两个状态都对：权限管理页该权限显示 **「已开通」** + 版本管理显示最新版 **「已发布」**。

修复后文件上传立通（`im/v1/files` 返回 `file_key` → 发 msgtype=file 消息）。

## 诊断口诀

- **文本能发、文件不能发** → 文件权限没在"应用身份"下生效（坑 1 或坑 2）。
- 想快速排除客户端因素：用 app_id/app_secret **现取一个全新 tenant_access_token**，同一 token 发文本成功、传文件 99991672 → 100% 是服务端权限未生效，去查"身份 tab + 版本发布"。

## 实例（deep-research-report，2026-06-02）

- 报告交付要把 .xlsx 自动推到飞书。bot 文本秒通，文件持续 99991672。
- 试过：清 lark-cli 缓存、绕过 lark-cli 直连官方 API 现取新 token——均同错。
- 用户"重新发布"后仍报错 → 看截图才发现 `im:resource` 加在了**用户身份权限** tab，**应用身份**下只有 3 条消息权限。
- 把 `im:resource:upload` 加到**应用身份** + 发版本审核通过后，一次成功（P2P 私聊按 email 直发，无需建群）。
