# 把浏览器自动化环境的假象当应用 bug

> 来源：three.js 世界模型 viewer"双击不下钻"排查（2026-07）
> 结局：一个真 bug（标签 sprite 不可点 + OrbitControls 吃双击）外面裹了四层环境假象，在假象里空转了多轮

## 问题

用自动化浏览器（Claude in Chrome / CDP / headless）验证 Web 页面时，**自动化环境本身会制造一批"复现失败"**。如果不先剥离这些假象，会把环境限制误诊为应用 bug，反复改一段本来没问题的代码。

## 限制矩阵（macOS Chrome 实测）

| 环境行为 | 制造的假象 | 对策 |
|---|---|---|
| 后台窗口 rAF 停转 | 动画不动、矩阵不更新→拾取/命中全失败 | 拾取前手动 `updateMatrixWorld(true)`；加 `setInterval` 兜底帧；把窗口拉到前台再测 |
| 后台 `setTimeout` 节流到 ≥1s | 依赖短定时器的交互（双击判定等）时序全乱 | 用事件时间戳判定，不用定时器 |
| CDP 注入的鼠标事件不产生原生 `dblclick` 等手势 | "双击没反应"——但真机是好的 | 用 `pointerup` 自己判双击（间隔<450ms、位移<8px）；同时给功能留非手势入口（按钮） |
| HTTP/file 缓存返回旧页面 | 改了代码"没生效" | URL 加 `?v=时间戳`；或响应头禁缓存 |
| headless Chrome WebGL 截图空白 | "渲染坏了" | WebGL 验证必须用有头浏览器 |
| 录屏类工具（gif_creator）在此场景 0 帧 | 无法交付操作演示 | 改为页面内置自动导览（脚本驱动相机+高亮），一个按钮触发 |

## 正确的排查顺序

1. **先证明事件到达了应用**：在事件处理器第一行 `console.log` 打探针，用自动化工具读 console——探针没打出来是环境问题，打出来了才是应用问题
2. **破缓存**（`?v=`）再谈"改了没生效"
3. 需要视觉确认的（WebGL/动画），**有头+前台**窗口测一次作为基线
4. 以上都排除后，剩下的才值得改代码

## 反模式信号

- 同一段交互代码改了三轮，每轮"还是不行"，但从没验证过事件是否真的派发到了页面
- 修复验证全程在后台标签页/headless 里做
- 把"自动化里录不出来"当成"功能不存在"，反之亦然
