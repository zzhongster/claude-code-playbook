# 反模式：日期时间字符串的两个隐蔽格式陷阱——isoformat 'T' vs SQLite 空格、strptime 未补零宽松解析

## 一句话结论

把时间当字符串比较或校验时，有两个各自独立、都能穿透常规测试的洞：①Python `datetime.isoformat()` 用 `'T'` 分隔日期与时间，SQLite `datetime('now')` 用空格——字节序上 `'T'`(0x54) > `' '`(0x20)，混用两种来源做字符串比较时**同一天的记录恒被判为"更晚"**，时间窗聚合边界日恒超算；②`datetime.strptime("2026-8-8", "%Y-%m-%d")` **成功解析**（%m/%d 不强制补零），用它当格式守卫拦不住未补零输入——而未补零字符串一旦进了字符串比较的库（如 `expires_at < today`），`"2026-8-8" 与 "2026-08-08"` 的逐字符比较结果是错乱的（`'8' > '0'`），可能导致 key 永不过期。

## 场景

- SQLite（或任何把时间存 TEXT 的库）+ Python 应用层混合写入时间戳，SQL 里用字符串比较做时间窗过滤
- CLI/配置接受 `YYYY-MM-DD` 自由输入，入库后靠字符串比较判过期/排序
- 用 `strptime` 做"格式校验"

## 踩坑经历

trade-agent-data S11 计费系统（2026-07-09）：

1. **isoformat vs datetime('now')**：BillingStore 的 `_now()` 初版用 `datetime.now(timezone.utc).isoformat()` 写 ledger.ts，而用量聚合 SQL 用 `ts >= datetime('now', '-7 days')` 比较。task reviewer 逐字节 repro：`"2026-07-09T…" > "2026-07-09 …"` 恒真——边界日的记录**永远**落进窗口，7 天用量恒多算一天。修复：`_now()` 改 `strftime("%Y-%m-%d %H:%M:%S")` 与 SQLite 输出逐字节同格式，并钉一条**格式契约测试**（断言 Python 侧产出与 `SELECT datetime('now')` 前 10 字节一致）。更深一层：修复后的第一版回归测试用的就是 SQLite 格式造数据，在旧 bug 代码上也能过（假 RED，另见 `tdd-fake-red.md`）。
2. **strptime 宽松**：终审要求给 CLI `--expires` 加格式校验（自由字符串直通字符串比较的库，`2026-8-8` 会错乱判定）。修复者先写 `strptime(x, "%Y-%m-%d")`，实测发现 `"2026-8-8"` **通过**——Python strptime 对未补零宽松。最终双闸：正则 `^\d{4}-\d{2}-\d{2}$` 强制补零 + strptime 验真实日历日（拦 `2026-02-30`/`2026-13-01`），都过才放行。

## 解法

1. **单一格式源**：凡时间入库做字符串比较，产出函数收敛到一处（如 `_now()`），格式与库函数输出（`datetime('now')`）逐字节对齐。
2. **格式契约测试**：不是测"窗口逻辑对不对"，而是直接断言两个来源的字符串格式相同——逻辑测试可能双方用同格式造数据而测不出混用。
3. **strptime 不是格式校验器**：它是解析器，宽松是特性。强格式校验 = 正则锁形态 + strptime 锁日历合法性，两道闸。
4. **模式识别**：SQL 里出现 `ts >= datetime('now'` 且应用层有 `isoformat()` 就报警；CLI 日期参数直通 TEXT 列字符串比较就报警。

## 教训

- 字符串比较时间"能跑"不等于"对"：格式不一致的错是**系统性偏移**（恒多算/恒不过期），不是随机崩溃，冒烟测不出来。
- 校验器的宽松语义要实测：`strptime`、`float()`（接受 `"inf"`，见 `int-float-cast-overflowerror.md`）这类标准库入口都比直觉宽松——写守卫前先拿恶意形态喂一口。
