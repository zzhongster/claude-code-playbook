# 反模式：调度循环无退避 + 终态记录卡在进度前沿 = 静默活锁

## 一句话结论

长驻调度器如果对"本轮没有实际推进"的返回**不做退避**，而进度游标又由一条**永不重试的终态记录**（ERROR/FAILED）钉死，就会进入静默活锁：进程活着、日志不写、CPU 和数据库空转，可以一转就是几个星期——监控上看"daemon 在跑"，实际什么都没干。

## 踩坑经历

2026-07 复查 novel-ingest-agent（小说批量提取 daemon）：nettop 发现它对 Postgres 拉着稳定 **17 MB/s** 流量，但 out.log 停在 6 月 3 日——**空转了六个星期**。

8 秒 SQL 全量日志还原出死循环，参数永远是同一个批次：

1. 批次 `0681-0690` 在 6 月 3 日因 Anthropic API 超时（重试 3 次放弃）进入 `ERROR` 态；
2. 每轮调度：`wiki_progress()` 扫实体文件算进度 → 最大章节停在 680（因为 0681-0690 从没成功入库）→ 调度器算出"下一批" = 0681-0690；
3. `run_batch` 加载后发现 `ERROR` 不匹配任何处理分支，**直接原样返回**；
4. `run_forever` 拿到非 None 返回值，认为"跑完一批"，**没有任何 sleep，立即开始下一轮**。

于是错误批次永不重试 → 进度永不推进 → 每秒空转两轮，每轮 `SELECT *` 白拉 2×3 MB 的 jsonb `state_data` 外加全量扫一遍 wiki 目录（2 次/秒 × 2 次 × 3 MB ≈ 17 MB/s，与 nettop 对上）。

同族变体（同月，万古神帝 kb 订阅版提取器）：撞订阅限额的报错措辞（"You've hit your session limit"）不在识别列表里，被当普通故障连续计数——那是反面方向的失配：**该长退避的被当短失败**。两个案例合起来说明：失败分类错一格，循环行为就错一个数量级。

## 为什么隐蔽

- 进程存活、PID 在，健康检查（如果只查存活）全绿；
- 日志一个字不写——因为"没有新东西可记"恰恰是症状本身；
- 唯一外显信号是**资源侧**的：稳定异常的 DB/CPU/网络流量。没人看 nettop 就永远发现不了。

## 修法

1. **调度循环对"没有实际推进"必须退避**：区分"完成了一批"（立即下一轮）与"无事可做/失败原样返回"（指数退避 sleep 或等唤醒事件）。判据用**副作用**（进度游标前移、产物落盘）而不是"函数返回了非 None"。
2. **终态必须有出路**：ERROR 态要么进重试队列（带上限与冷却），要么被调度器显式跳过（游标越过它继续后面的批次），绝不能既不重试又挡在进度前沿。断点锚设计（如"文件存在性"）天然免疫此问题——失败批次无产物，重跑自动补位。
3. **状态检查别拖大字段**：`SELECT state FROM batches WHERE id=$1`，不要 `SELECT *` 把 3 MB 的 jsonb 一起拉回来——活锁的放大器就是它。
4. **监控盯"进度速率"而非"进程存活"**：`最新产物时间戳落后当前时间超过 N 小时` 才是真报警条件。

## 通用判据

写任何 `while True` 调度器时问三个问题：

- 失败/空转分支睡多久？（答"不睡"= 事故预定）
- 一条记录永远处理不成功时，第二天调度器在干什么？（答"还在试它"或"卡住"= 活锁预定）
- 只看进程存活能发现它停摆吗？（答"不能"= 补进度速率监控）
