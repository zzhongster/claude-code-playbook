# 反模式：TDD 假 RED——测试失败的原因不是你要防的 bug

## 一句话结论

TDD 的 RED 只证明"测试当前失败"，不证明"测试失败在你要防的那条路径上"。写回归测试时如果造的数据/走的路径绕开了 bug 的触发条件，测试在**带 bug 的旧代码上也能通过**——绿灯是假安全。验证一条回归测试的唯一硬标准：**revert 修复、只留测试，跑一遍必须 FAIL**（revert-FAIL 证据）。

## 场景

- 修 bug 后补回归测试，测试数据用"修复后的正确形态"构造
- 测试断言的是下游效果（窗口、聚合、输出），而 bug 在上游（格式、来源、路径）
- 评审只看"测试existed + 全绿"，不核测试与 bug 触发条件的对应关系

## 踩坑经历

trade-agent-data S11 BillingStore（2026-07-09）：ledger 时间戳 bug 是 Python `isoformat()` 的 'T' 分隔符与 SQLite `datetime('now')` 空格格式混用，字符串比较边界日恒超算（详见 `datetime-string-format-traps.md`）。修复后 implementer 补的"回归测试"是：**用 SQLite 格式**造 ledger 行 → 断言 7 天窗口聚合正确。task reviewer 抓出致命点：这条测试造数据时就用了正确格式，等于绕开了 bug 的触发条件——把修复 revert 掉（`_now()` 改回 isoformat），这条测试**依然通过**，因为它测的窗口逻辑本来就没错，错的是 `_now()` 的产出格式。真正的回归测试是**格式契约测试**：直接断言 `_now()` 产出与 `SELECT datetime('now')` 同格式——revert 后立刻 FAIL。最终以"revert 修复→测试 FAIL→恢复修复→测试 PASS"的完整证据链关单。

## 解法

1. **revert-FAIL 验证**：回归测试写完后，暂存修复（`git stash` 或手工改回），确认测试 FAIL 且失败信息指向该 bug，再恢复。这一步 30 秒，是回归测试的"RED"。
2. **测 bug 的源头而非下游症状**：bug 在格式/来源/路径上时，断言直接打在源头（产出格式、调用参数），下游逻辑测试防不住"上游换了来源"。
3. **评审问一句**："这条测试在旧代码上会失败吗？为什么？"——答不出触发机制的回归测试按未覆盖处理。
4. **AI 流水线特化**：subagent 实现者天然倾向"造干净数据测干净逻辑"；给 fixer 的任务书里明确要求附 revert-FAIL 证据，而不只是"补测试"。

## 教训

- "测试 + 全绿"是流程证据不是覆盖证据；RED 的价值全在"失败原因正确"。
- 越是修复后补的测试越容易假——写测试的人满脑子是修复后的形态，天然用它造数据。先决定"旧代码会怎么错"，再倒推测试输入。
