# 反模式：碎片化多 Worker

## 一句话结论

把需要全局意识的任务拆成多个孤立 Worker 并行，会导致搜索偏科、判断失准、工具调用暴增但效果更差。

## 踩坑场景

"搜索太慢了，拆成 5 个 Worker 并行搜不同关键词，然后合并去重"

## 发生了什么

### 设计预期

```
Worker A: 搜索关键词 1-3 → 发现 10 家
Worker B: 搜索关键词 4-6 → 发现 10 家
Worker C: 搜索关键词 7-9 → 发现 10 家
→ 合并去重 → 25 家企业 → 筛选 → 保留 20 家
```

### 实际结果

```
Worker A: 搜索关键词 1-3 → 发现 15 家（偏向家具类）
Worker B: 搜索关键词 4-6 → 发现 12 家（偏向家具类，和 A 大量重复）
Worker C: 搜索关键词 7-9 → 发现 13 家（偏向设计师小店）
→ 合并去重 → 40 家
→ 筛选（孤立的 FilterWorker）→ 0 家保留 ❌
```

### 为什么会这样

1. **搜索偏科**：每个 Worker 不知道其他 Worker 搜到了什么。Worker A 搜到一堆家具店后不会想"缺了文具连锁，换个方向"——它只管自己的关键词
2. **重复浪费**：不同 Worker 用不同关键词搜到相同企业，309 次搜索产出 40 家，效率极低
3. **判断失准**：独立的 FilterWorker 只看到"企业数据 + 筛选规则"，没有调研全程的意图理解，把 description 中的"中小型"当成硬性标准
4. **质量反转**：工具调用 449 次 vs 单对话 30 次，成本更高但结果更差

## 根因

碎片化的本质问题是**上下文隔离**。每个 Worker 的上下文窗口里只有自己的 prompt 和工具调用历史，缺少：

- 其他 Worker 已经发现了什么（协作信息）
- 整体目标到底是什么（意图理解）
- 当前还缺什么（缺口意识）

这正是需要"全局意识"的部分，也是单对话模式的核心优势。

## 替代方案

### 方案一：单对话搜索 + 多 Worker enriching（推荐）

```
单对话: 搜索 + 去重 + 补盲区（需要全局意识）→ 20 家
多 Worker: 各自深度调研一家企业（独立无依赖）→ 20 家详细数据
单对话: 最终筛选和排序（需要全局意识）
```

### 方案二：多 Worker + 共享上下文

如果必须并行搜索，给每个 Worker 共享上下文：
- 其他 Worker 已发现的企业列表
- 全局的品类覆盖状态
- 明确的搜索方向分工（不要按关键词分，按品类/子任务分）

## 判断标准：该不该拆

| 问题 | 答 Yes → 可以拆 | 答 No → 不要拆 |
|------|---------------|--------------|
| 每个子任务是否完全独立？ | ✅ | ❌ |
| 子任务之间是否需要协调？ | ❌ | ✅ 不要拆 |
| 一个子任务的结果是否影响另一个的策略？ | ❌ | ✅ 不要拆 |
| 最终合并是否只是简单拼接？ | ✅ | ❌ |
