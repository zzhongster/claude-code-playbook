# 反模式：`2>/dev/null` 把"工具没装/命令失败"伪装成"真实的空数据"

## 一句话结论

诊断命令上随手加 `2>/dev/null`，会把 `command not found`、权限错误、参数错误一起吞掉，让"工具根本没跑"看起来像"跑了、结果是空/零"。于是你**基于一个从未产生的测量值下结论**，South 一路走偏。

## 真实案例

排查"VPS IP 是否被 GFW reset"，在 VPS 上抓包：

```bash
ssh vps 'timeout 35 tcpdump -i any -nn "tcp port 443" 2>/dev/null > /tmp/cap.txt'
# → "抓到 0 包"
```

**结论（错的）**：443 端口没流量 → 连接被秒 reset → IP 被封。

真相：**VPS 根本没装 tcpdump**。`2>/dev/null` 把 `tcpdump: command not found` 吞了，空文件被读成"0 包"。
幸好同时另一条命令显示"xray 有 90 个真实出站连接"——这个矛盾才暴露了"0 包"是假的。去掉 `2>/dev/null` 一看：

```
timeout: failed to run command 'tcpdump': No such file or directory
```

同一轮排查还踩了两个同类假象：
- `ss 'sport = :443'` 在某版本 ss 上过滤不生效、返回 0 → 误判"无连接"，换 `awk` 数才看到 53 个入站连接。
- `curl 域名` 在开着 fake-ip 的机器上被解析成 `198.18.x` 假地址而失败 → 误判"拉不到订阅"。

## 为什么危险

- 空输出 / 0 / 假地址**看起来像合法的测量结果**，不像错误，于是被当真。
- 在多组件系统里，一个假"0"会把根因指向错误的组件（这里：从"服务端正常"误导到"网络层被封"）。
- 越是"看起来印证了我的假设"的空结果，越要警惕——确认偏误会让你跳过验证。

## 正确做法

1. **诊断阶段别吞 stderr。** 想分离 stdout/stderr 就分文件，别 `2>/dev/null`：
   ```bash
   cmd 2>err.log >out.log   # 出问题先看 err.log
   ```
2. **先验证工具/前提存在**，再信它的输出：`command -v tcpdump || echo MISSING`。
3. **交叉验证矛盾**：当某个测量"恰好印证假设"时，找一个**独立**的观测量对账（这里是"90 个真实出站连接" vs "抓到 0 包"）。两个硬证据打架时，先怀疑测量，别急着下结论。
4. **区分"零"和"没测到"**：`0 包`、`空字符串`、`HTTP 000`、`返回 0 行`——先问"这是真的空，还是命令压根没成功跑"。

## 关联

- 同源教训："TCP 握手快 ≠ 隧道传输快"——`nc -z 443` 0ms 不代表节点能用，要测完整链路。测量手段本身的局限也是一种"假象"。
