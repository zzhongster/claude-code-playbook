# 模板查询预搜索/上下文填喂 — 给足上下文反而更贵更危险

> 来源：trade-ai 深度报告 #329 C 层实战（2026-07-16，实现→验证→当天下线）
> 实证：同任务三轮对照，预搜索版比不预搜索版**贵 11%、输入多 28%**，且敏感内容拦截率上升

## 反模式描述

直觉方案：Agent 每步要跑 5 条固定查询模板，那就在进入工具循环**之前**并行预跑这些查询，把结果注入首条消息——"模型从已有结果起步，省掉逐条搜索的往返轮数"。

听起来稳赚：省 3-5 轮全上下文调用。实测是**负优化**。

## 为什么翻车（两个机制）

### 1. 模型会把省下的预算再花掉
推理模型（glm-5.2 实测，其他模型大概率同样）面对轮数预算的行为是"花满为止"。预搜索没有减少它的工作量，只是**垫高了它的起点**——它拿着初步结果跑了更多衍生查询，最终输入 token 反而 +28%。

> 省钱的正解是给它"停止的纪律"（够了就停的规则），而不是"更高的起点"。

### 2. 预注入内容绕过了模型的自我审查
模型自己检索敏感内容时，会在组织输出的过程中做措辞软化；而**原始搜索摘要直接注入首条消息**等于把未经软化的敏感原文塞进输入——国产模型的输入侧内容过滤直接拒收（实测输入侧 1301 拦截从 0 涨到 2/3 次）。

## 修正后的原则

- 想省轮数 → 加"停止纪律"提示词（同域名不重复抓/无冲突不二次核对），实测 -7% 且步骤 7 拦截率下降
- 想给上下文 → 给**结构化结论**（前序步骤的 JSON 产物），不给原始检索物料
- 保留能力但下线接线：预搜索代码作为 Worker 可选参数保留（一行可重开），编排层不再默认使用

## 识别信号

你在做这件事的信号：prompt 里出现"以下是我帮你搜好的结果，请勿重复搜索"。先跑 A/B 再上——这个方向的直觉可靠性很低。
