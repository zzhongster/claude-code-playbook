# 反模式：多语言名称检索只查了一个语言字段——中文名零命中，英文/拼音名却能搜到

## 一句话结论

企业/人物/产品这类实体在数据库里往往同时存**多个语言的名字字段**（如 ES 的 `company_name_internat` 英文/拼音名 + `company_name_native` 中文/本地名）。检索若只 `match` 其中一个字段，用另一种语言的名字就**零命中**——最隐蔽的是英文/拼音名字段里存的是**音译拼音**（"Guang zhou nan fang ce hui"）而非意译，跟用户输入的中文（"广州南方测绘"）没有任何 token 重叠，match 必然 0，而系统看起来"在按名字搜"、不报错，用户只会以为"数据库里没有这家"。

## 场景

- 底层库同时有 internat/native、en/local、romanized/original 两套名称字段
- 检索工具/接口的 name 查询只写了一个字段：`{"match": {"company_name_internat": ...}}`
- 官方前端/后端界面查得到（它查了 native 字段），但新做的 API/MCP 工具查不到——两条查询路径字段覆盖不一致
- 音译拼音场景最坑：拼音字段与中文输入零 token 重叠，不是"排序靠后"而是"完全 0 命中"

## 踩坑经历

trade-agent-data（2026-07）：`search_companies` MCP 工具用中文"广州南方测绘"搜零结果，英文名能搜到。用户反馈"按理说数据库里有中文名"。直连 ES 复现，铁证：
- 只 match `company_name_internat`（当前工具查法）→ 中文名命中 **0**
- 改 match `company_name_native`（中文名字段）→ 命中 **303**（与官方前端界面完全一致）
- 样例：native="广州南方测绘仪器有限公司"，internat="Guang zhou nan fang ce hui yi qi..."（**拼音，非意译**）——这解释了为何中文输入对 internat 零 token 命中。

根因：`_build_dsl` 的 name 分支只覆盖英文/拼音字段，漏了中文字段；而官方界面查的是中文字段，所以"前端搜得到、工具搜不到"。

## 解法

1. **name 查询覆盖所有语言字段**：单 `match` 改 `multi_match` 跨字段：
   ```python
   {"multi_match": {"query": name,
                    "fields": ["company_name_internat", "company_name_native"],
                    "operator": "and"}}
   ```
2. **输出也带上另一语言名**：搜中文名却只返回一列拼音结果，用户还是认不出——响应补 `name_native`（有值才加），可读性和可搜性一起补齐。
3. **对齐权威查询路径**：有官方前端/后端"搜得到"时，去看它查了哪些字段，把工具的字段覆盖对齐过去——"前端搜得到、我的工具搜不到"几乎总是字段覆盖不一致。
4. **复现要连真库对比命中数**：`match 字段A` vs `match 字段B` 的 total 对比是最快铁证，别靠读代码猜。

## 教训

- "按名字搜"不是一个字段的事——实体有几种语言/写法的名字，检索就要覆盖几个字段，否则就是"只支持一部分语言"的隐性缺陷。
- **音译 vs 意译**决定了它是"0 命中"而非"排序靠后"：拼音字段和中文 query 无 token 交集，静默零结果，比部分命中更难被发现。
- 同一份数据的多条查询路径（前端 / 后端 / API / MCP 工具）字段覆盖要一致；不一致时以"能搜到的那条"为准对齐。
