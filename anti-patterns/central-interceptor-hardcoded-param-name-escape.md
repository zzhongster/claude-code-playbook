# 反模式：中央拦截层按硬编码参数名钳制，新工具换个计数参数名就静默逃逸

## 一句话结论

在 transport/gateway 这类"中央拦截层"里按**硬编码的参数名**（如 `limit`）做钳制/限流/校验时，任何用了不同参数名的新工具（`ref_ids`、`max_results`、`ids`、`top_n`）都会**静默绕过**这道防线——工具上声明的上限（`max_items_per_call=50`）形同虚设，因为拦截层根本不认它的计数参数。危险在于：声明存在、看起来有防护，但契约（声明）与实现（拦截逻辑）脱节，编译器不报、常规单测也测不到（测的是工具自身逻辑，不是"中央钳有没有覆盖到它"），只有人工逐个核对契约、或专门写"拦截层覆盖面"测试才抓得到。

## 场景

- 有一层中央 dispatch/interceptor（MCP `call_tool`、HTTP gateway、RPC 中间件），对所有下游统一施加某种约束
- 约束实现按具体字段名匹配：`if "limit" in args: args["limit"] = clamp(args["limit"], ...)`
- 新增工具的计数/分页参数没叫 `limit`，而叫 `ref_ids`/`max_results`/`page_size`
- 新工具的 spec 里还**声明**了上限（`max_items_per_call=50`），给人"已受保护"的错觉

## 踩坑经历

trade-agent-data（2026-07）：S9 曾做了 transport 中央 limit 钳制（`_clamp_limit` 按 `limit` 参数统一钳到 `max_items_per_call`）。S12 新增 `get_company_credit` 工具，入参是 `ref_ids`（批量公司 id）、声明 `max_items_per_call=50`。opus 整分支终审逐行 trace 后发现：中央钳只钳名为 `limit` 的参数，`ref_ids` 完全逃逸，声明的 50 从不生效——调用方传 500 个 ref_id 会构造 500 元素的 `IN (...)` 单查。影响被纯读/参数化/DB 超时兜底/扁平计费限制住，非红线破口，但属实打实的"声明-实现契约缺口"。**这已经是同一项目第二次踩**：早在 S6/S10 的 backlog 里就记过"计数参数命名硬编码 limit，新工具用别的名会静默逃逸中央钳"——记了 backlog 没根治，S12 就真长出来一个实例。修复：工具内自钳 `ids = ids[:50]` + 超限 warning + 常量注释点明"中央钳只认 limit，本工具须自钳"。

## 解法

1. **别按单一字段名钳**：中央拦截层要么用**元数据驱动**（工具在 ToolSpec 里声明"哪个参数是计数参数"，拦截层读元数据而非硬编码 `limit`），要么约定所有计数参数**统一命名**（全叫 `limit`）并在注册时校验。
2. **写"覆盖面"测试**：不要只测每个工具自身，要专门测"每个声明了 `max_items_per_call` 的工具，其计数参数确实被中央钳覆盖"——遍历注册表断言，新工具漏接立刻红。
3. **backlog 记了就根治**：这个坑第一次以 backlog 形式记录时没修，第二次就以真实缺陷长出来。可 grep 的机械检查（`max_items_per_call` 声明 vs 拦截层认的参数名）应在加新工具时跑，而非留 backlog。
4. **声明即契约**：ToolSpec 里写了 `max_items_per_call=50` 就是对调用方的承诺；承诺与实现脱节比"没有承诺"更危险（后者会让人自己加防护，前者让人以为有防护）。

## 教训

- "中央统一处理"的价值全在**覆盖完整**；一旦按具体名字匹配，覆盖就随命名漂移而漏，且漏得无声。
- 编译器/类型系统看不见"拦截层认不认这个参数名"——这类跨层契约缺口只有人工审契约或专门的覆盖面测试能抓。整分支终审逐层 trace 正是为了抓这种单任务评审看不到的涌现缺陷（另见 whole-branch-review-catches-emergent-defects）。
- backlog 里的"命名逃逸"风险不是提醒，是定时炸弹：不根治，下一个新工具就是引信。
